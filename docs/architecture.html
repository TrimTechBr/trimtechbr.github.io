Architecture & security

How AER collects data, renders the report, and stays strictly read-only and fully offline.

On this page

Module layout Data flow Processing layers Security & privacy

Module layout

AER is a dot-sourced PowerShell module with a clear separation between collection, orchestration, rendering and export:

AER.psd1 / AER.psm1          Module manifest + loader (dot-sources everything)
Public/Invoke-AerReport.ps1  Orchestrator (context → parallel collection → render)
Core/ResourceGraph.ps1       ARG query helper, columnar-safe row expander, ARM batch, aggregators
Collectors/*.ps1             One Get-Aer* function per domain → returns a plain object
Renderer/
  Dashboard.ps1              Page registry, sidebar nav, per-page HTML scaffolds, document shell
  Export.ps1                 Offline XLSX/PDF generation and native OpenXML/PDF helpers
  Assets/Script.ps1          The single-IIFE vanilla-JS app (tables, charts, diagrams)
  Assets/Style.ps1           The stylesheet

Data flow

The flow is linear and easy to reason about:

  1. Collectors query Azure (Resource Graph + ARM REST) and return PowerShell objects.
  2. The orchestrator runs collectors in parallel, then serializes their output into window.AerData.
  3. The renderer emits one HTML file per page plus shared assets/ (style.css, app.js, data.js).
  4. The exporter creates exports/aer-report.xlsx and exports/aer-report.pdf.
  5. In the browser, the JS reads AerData and builds the UI on the client — no server involved.
Azure (ARG + ARM)  →  Collectors  →  Orchestrator  →  window.AerData
                                                  ├→ Renderer  → HTML + assets/
                                                  └→ Exporter  → exports/ (xlsx, pdf)

Processing layers

LayerPurpose
CollectionRetrieve Azure resource, monitoring, governance and security configuration data using read-only calls.
ProcessingNormalize findings into a consistent reporting model and compute aggregates/KPIs.
PresentationGenerate readable HTML, XLSX and PDF outputs for technical and management audiences.

Security & privacy

  • Read-only by design. AER only issues read/list/query calls (Resource Graph + ARM GET / POST …/summarize|queryResults|list). It performs no write operations.
  • Your data stays with you. Collection happens locally with your credentials; the resulting HTML embeds the data in assets/data.js. Nothing is uploaded anywhere.
  • Offline. The generated report has no external dependencies and works under file://.
Treat output as sensitive. The report contains environment metadata (resource names, IDs, configuration). Treat the output folder as you would any infrastructure inventory.
